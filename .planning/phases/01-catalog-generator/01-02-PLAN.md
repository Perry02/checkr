**TGT-01: gen_skip_program** — Guarantee `Command::Skip` appears.

```rust
/// TGT-01: Guaranteed `skip` command.
///
/// Always emits `Command::Skip`. Optionally wraps it with a random
/// assignment before and/or after for variety.
fn gen_skip_program<R: Rng>(cx: &mut CompilerContext, rng: &mut R) -> Commands {
    let mut cmds: Vec<Command> = Vec::new();
    // Optional prefix assignment
    if rng.random_bool(0.5) {
        cmds.push(Command::Assignment(gen_target(cx, rng), gen_aexpr(cx, rng)));
    }
    // GUARANTEED skip
    cmds.push(Command::Skip);
    // Optional suffix assignment
    if rng.random_bool(0.3) {
        cmds.push(Command::Assignment(gen_target(cx, rng), gen_aexpr(cx, rng)));
    }
    Commands(cmds)
}
```

**TGT-04: gen_simple_do** — Guarantee `Command::Loop` with exactly 1 guard.

The program graph for `Command::Loop` automatically emits a loop-done edge whose condition is the negation of all guard conditions. With 1 guard, `done = !b`. This exercises the basic loop-done computation in student compilers.

```rust
/// TGT-04: Single-guard `do b -> cmds od`.
///
/// The program graph emits a loop-done edge with `done = !b`. This exercises
/// the simplest loop termination case.
fn gen_simple_do<R: Rng>(cx: &mut CompilerContext, rng: &mut R) -> Commands {
    let body = Commands(vec![
        Command::Assignment(gen_target(cx, rng), gen_aexpr(cx, rng)),
    ]);
    let guard = Guard(gen_bexpr(cx, rng), body);
    Commands(vec![Command::Loop(vec![guard])])
}
```

**TGT-05: gen_do_n_guards** — Guarantee `Command::Loop` with exactly `n` guards (n=2,3,4).

With n guards, the done-expression is `!b1 && !b2 && ... && !bn`. This targets the `donegc` computation — the conjunction of negated guard conditions that determines loop termination. More guards = more complex done-expression = harder for buggy compilers to get right.

```rust
/// TGT-05: Multi-guard `do` with exactly `n` guards (n=2,3,4).
///
/// The program graph computes `done = !b1 && !b2 && ... && !bn`.
/// With n=4, this is a 4-way conjunction of negated conditions — a key
/// edge case for the `donegc` computation in student compilers.
fn gen_do_n_guards<R: Rng>(cx: &mut CompilerContext, rng: &mut R, n: usize) -> Commands {
    let guards: Vec<Guard> = (0..n)
        .map(|_| {
            let body = Commands(vec![
                Command::Assignment(gen_target(cx, rng), gen_aexpr(cx, rng)),
            ]);
            Guard(gen_bexpr(cx, rng), body)
        })
        .collect();
    Commands(vec![Command::Loop(guards)])
}
```

**TGT-06: gen_nested_if_in_do** — Guarantee an `if` inside a `do`.

Structure: `do b_outer -> (if b_inner -> cmd fi) od`

The outer construct is `Command::Loop`, and its guard's body contains `Command::If`. This exercises the compiler's handling of nested control flow where an if-node set appears inside a loop back-edge.

```rust
/// TGT-06: `if` nested inside `do` — `do b -> (if b2 -> cmd fi) od`.
///
/// The outer loop guard's body contains a `Command::If`. This exercises
/// nested program graph construction where if-edges are inside a loop cycle.
fn gen_nested_if_in_do<R: Rng>(cx: &mut CompilerContext, rng: &mut R) -> Commands {
    // Inner: if b_inner -> assignment fi
    let inner_body = Commands(vec![
        Command::Assignment(gen_target(cx, rng), gen_aexpr(cx, rng)),
    ]);
    let inner_guard = Guard(gen_bexpr(cx, rng), inner_body);
    let inner_if = Command::If(vec![inner_guard]);

    // Outer: do b_outer -> inner_if od
    let outer_body = Commands(vec![inner_if]);
    let outer_guard = Guard(gen_bexpr(cx, rng), outer_body);
    Commands(vec![Command::Loop(vec![outer_guard])])
}
```
